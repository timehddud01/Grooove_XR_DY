# 음악 인식 기능 실행 순서

## 1. 목적

DisplayAccess 앱에 휴대폰 마이크와 ACRCloud를 이용한 음악 인식 기능을 구현한다. TASK-01부터 TASK-05까지 순서대로 실행하고 각 단계의 코드, 테스트와 checklist를 함께 완료한다. 실행하지 않은 실기기 항목은 완료로 간주하지 않는다.

## 2. 구현 범위

- DisplayAccess 앱의 독립 Music 탭에서 휴대폰 마이크로 음악을 녹음한다.
- 녹음한 WAV를 Node 프록시를 통해 ACRCloud에 전달한다.
- 결과, 미인식, 오류, 취소 상태를 Android UI에 표시한다.
- 음악용 `DisplayGateway`의 실제 안경 출력은 구현하지 않았으며 앱은 `NoOpDisplayGateway`를 사용한다.

### 후속 범위

- Music 탭은 기존 DAT Display 샘플과 같은 앱에 있지만 음악 인식에는 안경 마이크나 MockDeviceKit 카메라/미디어를 사용하지 않는다.
- 음악 인식 결과를 실제 Meta 안경 Display에 렌더링하는 gateway는 아직 없다.
- AudD adapter 재도입, ShazamKit adapter, MockDevice 음악 통합은 이번 구현 범위가 아니다.
- 공급자별 200곡 비교 실험도 현재 구현 요구가 아니라 후속 검증 후보다.
- `RecognitionProvider.SHAZAM_KIT` enum 값은 확장 지점일 뿐 구현된 공급자를 뜻하지 않는다.

## 3. 의사결정 기록

초기 계획은 AudD API 기반 휴대폰 PoC였으나 실제 사용 중 인식 성능과 여러 오류가 확인되었다. 최종 구현은 앱에 공급자 비밀을 넣지 않고, 자체 Node 프록시가 ACRCloud를 호출하는 구조로 전환했다. 공급자는 ACRCloud를 사용하며 AudD 코드는 만들지 않는다.

ACRCloud 전환 뒤에도 도메인 인터페이스는 공급자와 분리했다. Android는 `MusicRecognizer`만 호출하며 ACRCloud 인증, 서명, 제한 정책은 서버가 소유한다. 이 경계는 공급자 교체 가능성과 secret 보호를 위한 필수 설계다.

## 5. 도메인 계약

### 포트

- `AudioInput.capture(durationMillis)`는 `Captured(AudioSample)`, `Failure(error)`, `Cancelled` 중 하나를 반환하고 `cancel()`로 활성 녹음을 중단한다.
- `MusicRecognizer.recognize(sample)`는 `Match(result)`, `NoMatch`, `Failure(error)` 중 하나를 반환한다.
- `DisplayGateway.present(state)`는 UI 상태를 외부 표시 장치로 전달하는 확장점이다. 현재 production 조립은 `NoOpDisplayGateway`이므로 안경 출력이 발생하지 않는다.

### 모델과 상태

- `AudioSample`: little-endian PCM 16-bit byte array, sample rate, 기본 mono channel. 길이로 `durationMillis`를 계산한다.
- `RecognitionResult`: 필수 `title`, `artist`; 선택 `album`, `artworkUrl`; 공급자를 나타내는 `RecognitionProvider` enum 필드. 현재 실제 `AcrCloudProxyMusicRecognizer` adapter가 이 필드에 `ACRCLOUD`를 설정한다.
- `InvalidAudioReason`: `EMPTY`, `TOO_SHORT`, `SILENT`, `DISTORTED`, `UNSUPPORTED_FORMAT`.
- `MusicRecognitionError`: `PermissionDenied`, `Timeout`, `Authentication`, `RateLimited`, `Network`, `InvalidAudio`, `AudioCapture`, `Provider`, `Unexpected`.
- `MusicRecognitionUiState`: `Idle`, `Listening`, `Recognizing`, `Matched`, `NoMatch`, `Error`이며 대응 status는 `IDLE`부터 `ERROR`까지 일대일이다.

### 정상 상태 흐름

`Idle → Listening → Recognizing → Matched|NoMatch|Error` 순서다. 녹음 실패는 `Listening → Error`, 오디오 품질 실패도 공급자 호출 없이 `Error`가 된다. 다시 인식할 때 이전 샘플이나 결과를 캐시하지 않고 반드시 새 12초 녹음을 시작한다.

취소 시 `requestGeneration`을 증가시키고 coroutine job과 `AudioInput`을 취소한 뒤 즉시 `Idle`로 만든다. 이전 generation의 늦은 응답은 `updateState`에서 무시한다. Music route가 dispose되거나 ViewModel이 clear될 때도 취소한다.

## 6. 오디오 캡처와 WAV 계약

### 캡처

- `PhoneAudioInput`은 휴대폰 `MediaRecorder.AudioSource.MIC`만 사용한다.
- 형식은 PCM 16-bit little-endian mono다.
- sample rate는 `44100 → 48000 → 16000 → 8000` 순서로 `AudioRecord.getMinBufferSize`가 지원하는 첫 값을 선택한다.
- ViewModel의 기본 녹음 길이는 12,000ms다.
- 첫 녹음 RMS가 128 미만이면 사용자 조작 없이 12,000ms 한 번만 더 녹음한다. 두 번째도 무음이면 `InvalidAudio(SILENT)`를 반환한다.
- buffer는 최소 buffer size와 4,096 중 큰 값을 사용하며 목표 byte 수까지 blocking read한다.
- 성공 시 duration, sample rate, RMS, peak, byte 수만 기록한다. 원본 오디오나 사용자 식별정보를 로그에 남기지 않는다.
- 성공, 실패, 예외, 취소 모든 경로에서 `stop()`을 시도하고 `release()`를 호출한다.

### 검증

`AudioSampleValidator` 기본 정책은 다음과 같으며 공급자 호출 전에 적용한다.

- 빈 byte array: `EMPTY`
- byte 수가 홀수: `UNSUPPORTED_FORMAT`
- 길이 3,000ms 미만: `TOO_SHORT`
- RMS 128 미만: `SILENT`
- 절대 진폭 32,700 이상인 sample 비율이 5% 초과: `DISTORTED`

### WAV 인코딩

`WavEncoder`는 PCM 앞에 정확히 44-byte RIFF/WAVE header를 붙인다. format은 PCM(1), channel/sample rate는 `AudioSample` 값, bits per sample은 16이며 byte rate와 block align을 계산한다. `data` chunk size는 PCM byte 수이고 결과 크기는 항상 `44 + pcm.size`다.

## 7. Android→프록시 계약

`AcrCloudProxyMusicRecognizer`는 `HttpURLConnection`으로 다음 요청을 보낸다.

```http
POST /v1/music-recognitions
Content-Type: multipart/form-data; boundary=...
X-User-Id: <설치별 UUID>

audio: filename="capture.wav", Content-Type="audio/wav", WAV bytes
```

- connect/read timeout은 각각 20초다.
- `X-User-Id`는 `music_recognition` SharedPreferences의 `anonymous_user_id`에 최초 생성한 UUID를 재사용한다.
- URL은 `BuildConfig.MUSIC_RECOGNITION_PROXY_URL`이며 `local.properties`의 `music_recognition_proxy_url`로 덮어쓴다.
- 기본값 `http://10.0.2.2:8787`은 Android emulator에서 개발 PC의 프록시에 접근하기 위한 값이다.
- 개발 중 HTTP를 위해 manifest에 cleartext가 허용되어 있다. production 배포에서는 HTTPS URL을 사용하고 cleartext 허용을 제거하거나 debug 전용으로 제한한다.
- 앱에는 `ACRCLOUD_ACCESS_SECRET`이나 access key를 넣지 않는다.

성공 응답은 아래 두 형태만 허용한다.

```json
{"status":"NO_MATCH","result":null}
```

```json
{"status":"MATCH","result":{"title":"...","artist":"...","album":null,"artworkUrl":null,"provider":"ACRCLOUD"}}
```

실패 응답은 `{"error":{"type":"...","message":"..."}}`다. 앱은 401/403 또는 `AUTHENTICATION`, 408/504 또는 `TIMEOUT`, 429 또는 `RATE_LIMITED`를 typed error로 변환하고 429의 `Retry-After` 초를 보존한다. socket timeout은 `Timeout`, I/O 예외는 `Network`, 그 외 예외는 `Unexpected`다.

## 8. 프록시와 ACRCloud 계약

### 런타임과 설정

프록시는 Node.js 20 이상에서 외부 npm runtime dependency 없이 `node:http`, `fetch`, `FormData`, `Blob`, `node:crypto`로 동작한다. 시작 시 다음 환경변수가 모두 필요하다.

- `ACRCLOUD_HOST`: scheme 없이도 허용하며 내부에서 정규화
- `ACRCLOUD_ACCESS_KEY`
- `ACRCLOUD_ACCESS_SECRET`
- 선택 `PORT`, 기본 8787

`music-recognition-proxy/.env.example`에는 legacy `AUDD_API_TOKEN`만 남아 있어 실제 ACRCloud 런타임과 불일치한다. 자동 테스트 통과 여부와 별개인 설정 문서 결함이므로 정리해야 한다. 예시 파일은 실제 secret 없이 `ACRCLOUD_HOST`, `ACRCLOUD_ACCESS_KEY`, `ACRCLOUD_ACCESS_SECRET`, `PORT=8787` 형식으로 맞추고, `music-recognition-proxy/.gitignore`의 `.env` 제외 규칙은 유지해야 한다.

### 입력 경계

- 정확히 `POST /v1/music-recognitions`만 처리한다.
- `X-User-Id`는 영문·숫자·하이픈 8~128자여야 한다.
- multipart의 `audio` field와 `audio/wav`, `audio/x-wav`, `audio/vnd.wave`만 허용한다.
- 파일 최대 5MiB, 전체 request 최대 `5MiB + 16KiB`다. 선언된 길이와 streaming 누적 길이를 모두 검사한다.
- RIFF/WAVE, PCM format 1, 16-bit, byte rate, data size와 44-byte 최소 header를 검사한다.
- WAV 길이는 0초 초과 12.5초 이하여야 한다.
- 설치 UUID별 분당 5회 제한은 process memory의 Map으로 관리하고 초과 시 429와 `Retry-After`를 반환한다. 서버 재시작·다중 instance를 넘는 영속 제한은 아니다.
- 업로드 오디오와 인식 결과를 저장하지 않고 secret·원본 audio·사용자 식별값을 로그에 기록하지 않는다.

### ACRCloud 요청 서명

프록시는 `https://{ACRCLOUD_HOST}/v1/identify`로 sample, sample_bytes, access_key, data_type=`audio`, signature_version=`1`, signature, Unix timestamp를 multipart 전송한다. HMAC-SHA1 canonical string은 정확히 다음과 같다.

```text
POST\n/v1/identify\n{access_key}\naudio\n1\n{timestamp}
```

서명은 access secret을 key로 UTF-8 canonical string을 HMAC-SHA1 처리한 Base64 값이다. upstream timeout은 20초다.

### provider 상태와 후보 선택

- ACRCloud status 0: 후보 선택 후 `MATCH`; 유효 후보가 없으면 `NO_MATCH`
- status 1001: HTTP 200 `NO_MATCH`
- status 3001/3014 또는 upstream HTTP 401/403: `AUTHENTICATION`
- status 3003/3015 또는 upstream HTTP 429: `RATE_LIMITED`
- timeout/abort: HTTP 504 `TIMEOUT`
- 다른 provider 오류: HTTP 502 `PROVIDER_ERROR` 또는 `UPSTREAM_UNAVAILABLE`
- title/artist가 빠진 선택 결과: HTTP 502 `INVALID_UPSTREAM_RESPONSE`

후보는 title과 첫 artist 이름이 모두 있는 항목만 남기고 ACRCloud `score` 내림차순으로 고른다. score 동률이면 metadata quality를 사용한다.

| metadata | 가점 |
|---|---:|
| ISRC | 40 |
| Spotify track ID | 15 |
| Deezer track ID | 10 |
| MusicBrainz track ID | 8 |
| YouTube video ID | 5 |
| label | 3 |

artist가 placeholder(`unknown`, `various artists`, `v.a` 및 코드에 정의된 깨진 placeholder 문자열)면 100점을 감점한다. 최종 응답은 첫 artist, album 이름을 정규화하고 현재 artwork URL은 `null`로 둔다.

## 9. 앱 UI와 권한 계약

- `AndroidManifest.xml`에 `RECORD_AUDIO`와 `INTERNET`를 선언한다.
- Music 탭은 DAT Display 연결 여부와 무관하게 활성화된 독립 route다.
- 인식 버튼을 누르면 Activity Result API로 마이크 권한을 확인·요청한다. 거절 시 공급자를 호출하지 않고 `PermissionDenied`를 표시한다.
- `Listening`과 `Recognizing`에서는 progress와 취소 버튼을 보이며 중복 인식 버튼을 막는다.
- `Matched`에는 제목과 아티스트, `NoMatch`와 각 error에는 재시도 가능한 안내를 표시한다.
- 재시도는 언제나 새 녹음이다. 결과/audio cache는 없다.
- `MusicRecognitionScreen.kt`의 한국어 문자열은 정상 UTF-8로 유지하고 UI에서 올바르게 표시되는지 확인한다.

## 10. 실행 순서

1. TASK-01: 도메인 계약과 테스트 골격
2. TASK-02: 휴대폰 녹음, 품질 검증, WAV 변환
3. TASK-03: Node ACRCloud 프록시와 Android 어댑터
4. TASK-04: UI, 앱 통합, 자동 검증
5. TASK-05: 운영 설정, 실기기 E2E, 최종 범위 확정

각 TASK는 대응하는 `docs/checklist`를 함께 확인하고, 선행 TASK의 완료 조건을 만족한 뒤 다음 TASK로 진행한다.

### TASK-01 — 도메인 설계와 테스트 골격

목적은 Android framework, ACRCloud, DAT에 의존하지 않는 음악 인식 핵심 계약을 먼저 고정하는 것이다.

1. `MusicContracts.kt`와 `MusicModels.kt`를 생성한다.
2. `AudioSampleValidator.kt`에 3초/RMS/clipping/format 정책을 구현한다.
3. `MusicRecognitionViewModel.kt`에 validator 주입, 상태 전환, 중복 실행 방지, generation 기반 취소를 구현한다.
4. version catalog에 JUnit 4.13.2와 kotlinx-coroutines-test 1.10.2를 선언하고 app module의 `testImplementation`에 연결한다.
5. `TestDoubles.kt`, `TestAudioFixtures.kt`, `MusicRecognitionViewModelTest.kt`를 생성한다.
6. match, no-match, capture/validation/provider failure, 취소 후 늦은 결과 무시, 재시도 시 새 capture를 테스트한다.

산출물은 세 port, typed model/outcome/error/UI state, 공급자 호출 전 validator와 Android device API·네트워크 독립 ViewModel 테스트다. 완료 조건은 UI·네트워크 구현 전에 핵심 흐름과 invalid audio 차단이 fake로 결정적으로 검증되고 `DisplayGateway` 기본값이 NoOp임을 명시하는 것이다.

### TASK-02 — 휴대폰 녹음·검증·WAV

1. `PhoneAudioInput.kt`에 12초 PCM16 mono 캡처, sample-rate fallback, 무음 1회 재녹음, 취소와 자원 해제를 구현한다.
2. TASK-01의 `AudioSampleValidator` 정책을 실제 캡처 sample에 적용하고 validation failure가 공급자 호출 전에 차단되는지 검증한다.
3. `WavEncoder.kt`와 `WavEncoderTest.kt`로 정확한 44-byte WAV header와 PCM 보존을 검증한다.
4. manifest에 `RECORD_AUDIO`를 추가한다.

산출물은 공급자와 무관한 유효 WAV sample이다. 완료 조건은 모든 terminal path에서 `AudioRecord`가 해제되고 잘못된 sample이 네트워크 단계에 도달하지 않으며 WAV 단위 테스트가 통과하는 것이다.

### TASK-03 — Node ACRCloud 프록시와 앱 adapter

1. `music-recognition-proxy/package.json`과 `src/server.mjs`를 생성한다.
2. 서버 환경변수 검증, endpoint routing, multipart parsing, request/WAV 제한과 설치별 rate limit을 구현한다.
3. `music-recognition-proxy/.env.example`을 `ACRCLOUD_HOST`, `ACRCLOUD_ACCESS_KEY`, `ACRCLOUD_ACCESS_SECRET`, `PORT` 예시로 갱신하고 `.gitignore`의 `.env` 제외를 유지한다.
4. ACRCloud `/v1/identify` canonical HMAC-SHA1 서명과 multipart upstream 요청을 구현한다.
5. provider status/HTTP/timeout을 공통 JSON 계약으로 변환하고 후보 score 및 metadata 품질 tie-break를 구현한다.
6. `test/server.test.mjs`에 서명·대표 후보, no-match, MIME/길이 거부, rate limit, 인증·quota·timeout 테스트를 작성한다.
7. Android의 `AcrCloudProxyMusicRecognizer.kt`에서 WAV multipart 요청과 typed response/error mapping을 구현한다.
8. `app/build.gradle.kts`에 local property 기반 proxy URL BuildConfig field를 추가한다.

산출물은 secret을 소유하는 zero-dependency 프록시와 secret을 모르는 Android adapter다. 완료 조건은 proxy 테스트 5개가 통과하고 request/response 및 오류 계약이 양쪽에서 일치하는 것이다. 과거 AudD token이나 adapter는 이 단계에 포함하지 않는다.

### TASK-04 — UI·앱 통합·자동 검증

1. `MusicRecognitionViewModelFactory.kt`에서 `PhoneAudioInput`과 `AcrCloudProxyMusicRecognizer`를 조립하고 설치별 UUID를 만든다.
2. `MusicRecognitionScreen.kt`에 runtime 마이크 권한, 상태별 UI, 취소와 재시도를 연결한다.
3. `AppScaffold.kt`에 독립 Music route/tab을 추가하고 route dispose 때 인식을 취소한다.
4. manifest에 `INTERNET`와 개발용 cleartext 설정을 반영한다.
5. `DisplayAccess/README.md`에 Music 기능, proxy URL 설정, emulator/실기기 차이와 권한·실행 절차를 추가한다.
6. 화면 문자열은 UTF-8 한국어로 확인하고 사용자에게 typed error별 복구 가능한 안내를 제공한다.
7. Android unit test와 lint, proxy test를 모두 실행한다.

산출물은 앱에서 실제 휴대폰 녹음과 로컬 프록시 호출까지 이어지는 Music 탭이다. 완료 조건은 화면 이탈/취소 후 stale 결과가 나타나지 않고 권한 거부와 모든 상태가 UI에 표현되며 자동 검증이 통과하는 것이다. 안경 Display 연결은 이 TASK의 완료 조건이 아니다.

### TASK-05 — 운영 설정과 실기기 E2E

`docs/tasks/TASK-05-운영-설정과-실기기-E2E.md`와 대응 checklist에 따라 실제 proxy credentials, device URL, 수동 E2E 증거와 운영 보안을 확인한다.

1. Node 20 이상 환경에 `ACRCLOUD_HOST`, `ACRCLOUD_ACCESS_KEY`, `ACRCLOUD_ACCESS_SECRET`을 설정하고 프록시를 실행한다.
2. emulator는 기본 `10.0.2.2:8787`, 실기기는 접근 가능한 개발 PC 주소 또는 배포 HTTPS URL을 `music_recognition_proxy_url`에 설정한다.
3. 실제 휴대폰에서 마이크 권한 허용/거부, 정상 곡 여러 개, 무음, 과입력, no-match, 네트워크 단절, timeout, 인증 실패, 429, 취소와 재시도를 검증한다.
4. 로그·서버 저장소·APK에 secret이나 원본 audio가 남지 않는지 점검한다.
5. HTTPS와 process 외부 rate limiting/관측성/배포 보안은 운영 환경에 맞춰 보강한다.
6. AudD/Shazam/200곡 공급자 비교와 실안경 음악 출력은 별도 후속 TASK로만 제안하고 현재 완료 범위에 섞지 않는다.

산출물은 환경별 실행 설정, E2E 결과와 명확한 범위 선언이다. 자동 검증은 완료됐지만 실제 휴대폰 다곡 및 장애 시나리오는 미검증이다. 따라서 해당 checklist는 실행 증거가 생기기 전까지 체크하지 않는다.

## 11. 검증 명령과 현재 기준선

PowerShell에서 다음 순서로 실행한다.

```powershell
Set-Location DisplayAccess
.\gradlew.bat test
.\gradlew.bat lint
Set-Location ..\music-recognition-proxy
npm test
```

현재 확인된 기준선은 Android unit test가 build variant별 11개 통과, proxy test가 5개 통과, Android lint 성공이다. 이 숫자가 달라지면 테스트 추가·variant 변경인지 회귀인지 확인해 기록한다. 자동 테스트는 실제 마이크, 네트워크와 ACRCloud 계정을 대체하지 않으므로 실기기 검증을 별도로 수행한다.

각 TASK를 실행한 뒤 대응하는 `docs/checklist/TASK-0N-*.md`에서 코드 존재만이 아니라 동작과 테스트 증거를 확인한다. 미실행 항목을 추정으로 체크하지 않는다.

## 12. 완료 정의

문서에 지정된 파일, 계약, 상태 흐름, 보안 경계와 테스트를 구현하고 자동 검증을 통과해야 한다. 실제 휴대폰 다곡 인식과 실패 시나리오는 별도 E2E 검증으로 완료 여부를 기록한다.

전체 완료는 다음 조건을 모두 만족할 때 선언한다.

- TASK-01부터 TASK-05까지 순서를 유지하고 각 선행 산출물을 다음 단계가 사용한다.
- 12초 휴대폰 PCM16 mono 캡처, 무음 1회 재녹음, 입력 검증과 44-byte WAV가 명세와 일치한다.
- Android→proxy→ACRCloud 계약, HMAC 서명, 후보 선택, 오류 변환과 제한 정책이 일치한다.
- Music 탭의 권한·상태·취소·재시도 흐름이 동작하고 취소된 요청의 늦은 결과가 차단된다.
- ACRCloud secret이 Android artifact에 없고 proxy에서만 환경변수로 주입된다.
- Android test/lint와 proxy test가 통과한다.
- 실제 휴대폰 E2E 결과를 성공·실패 모두 기록한다. 실행하지 않았다면 전체 운영 검증은 미완료로 표시한다.
- 현재 Music 기능이 `NoOpDisplayGateway`를 사용하여 안경에 결과를 출력하지 않는다는 범위를 문서와 checklist에서 동일하게 유지한다.

이 정의를 충족하기 전에는 기존 DisplayAccess의 DAT 연결·표시 기능이 정상이라는 이유로 음악 인식의 실안경 종단 연결이 완료됐다고 기록하지 않는다.
